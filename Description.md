# GitとGitHubの使い方を丁寧に見やすく解説

## 初期設定

### Git

最初に一度だけ設定。Gitの設定に関しては git config というコマンドを利用する。—global オプションを付けることで、ホームディレクトリの .gitconfig というファイルに設定が保存されていく。設定した後に、逐一 `git config 設定項目名` を実行して確認するとよい。

- `git config --list`
    - 現在のGitの**設定一覧を表示**。
- `git config --global user.name "Your Name"`
    - コミットに表示される**名前**を設定。
- `git config --global user.email "your_email@example.com"`
    - コミットに表示される**メールアドレス**を設定。
- `git config --global core.editor "code --wait"`
    - `code`はVS Codeを起動するコマンド。VS Codeインストール時にcodeという名前でpathに登録されているはず。
    - `-wait`オプションは、VS Codeでファイルを開いた後、その**ファイルを閉じるまでGitの処理を待機させる**ために必要。これがないと、エディタが起動した瞬間にGitが「編集が終わった」と勘違いしてしまい、コミットメッセージなどが空になってしまう。
    - つまり「**今後、Gitでエディタを使う時はいつでもVS Codeを使い、ファイルが閉じられるまで待ってください**」という設定をPC全体に適用するコマンド。
- `git config --list`
    - ちゃんと設定できてるか確認。
- `git config --global init.defaultBranch main`
    - デフォルトブランチの名前をmainにする。昔はmasterが主流だった。

設定項目を削除したいときは以下を実行。

- `git config --global --unset 設定項目名`

### GitHub

手元のパソコンとGithubとで通信してデータをやりとりするには、認証という手続きが必要になる。Githubで使用できる認証方法は２つあり（2025年時点）、「HTTPというプロトコルとパーソナルアクセストークンを用いたユーザー認証」または「SSHというプロトコルと公開鍵を用いたサーバー認証」である。いづれの方法も最初にコマンドラインやブラウザー上での設定が必要。SSHを使う場合は、最初の設定以降はコマンドライン上で認証のために、ユーザー名やパーソナルアクセストークンなどを入力する必要がないので、Githubに接続する際の手順が軽減できる。

公開鍵による認証

ローカルでペアの秘密鍵と公開鍵を作成し、公開鍵をGithunに登録する。鍵を用いて暗号化通信を行う。
まず鍵を作成する。作業場はCUIで、ディレクトリはホームディレクトリ(~)。

- `ssh-keygen -t ed25519 -C "sample@example.com"`
- コマンドの解説
    
    このコマンドは、GitHubなどのサーバーに安全に接続するための**「SSH鍵（秘密鍵と公開鍵のペア）」を作成する**ためのものです。
    
    現在、セキュリティとパフォーマンスの両面で**最も推奨されている設定**のひとつです。
    
    ---
    
    ### **コマンドの各部分の意味**
    
    - **`ssh-keygen`**
        - SSHの鍵ペアを作成（生成）するためのプログラムを起動します。
    - **`t ed25519`**
        - 作成する鍵の種類（アルゴリズム）を指定します。**Ed25519**は、従来のRSAよりも高速で、短い鍵長でも非常に高いセキュリティを誇る最新の形式です。
    - **`C "example@gmail.com"`**
        - 鍵に「コメント（Comment）」を付けます。一般的には自分のメールアドレスを入力し、**「誰の、どのための鍵か」を識別しやすくする**ために使われます。
    
    ### **生成されるファイル**
    
    成功すると、指定した場所に**2つのファイル**がセットで作成されます。
    
    - `id_ed25519` （**秘密鍵**: 自分だけの印鑑。**絶対に見せてはいけない**）
    - `id_ed25519.pub` （**公開鍵**: 相手に渡す鍵。GitHubの設定画面などに中身を貼り付ける）
    
    この鍵を作成したあとは、GitHubなどの設定画面に「公開鍵（.pubの方）」を登録することで、パスワード入力を省略して安全に通信ができるようになります。
    
- 生成した鍵の保存先の絶対パスが表示されるのでEnter
- パスフレーズ（パスワードみたいなもの）を入力する。今生成するSHH Key専用のパスフレーズの設定のため、GitHubとは別の新たなものを入力する。入力しても画面に表示されないことと、２回入力が必要なことに注意。

生成した公開鍵をGitHubの設定画面に登録する。次のコマンドを実行してクリップボードにコピーする。

- `clip < /c/Users/ユーザー名/.ssh/id_ed25519.pub`
    - macOS → `pbcopy < /c/Users/ユーザー名/.ssh/id_ed25519.pub`
- GitHubの [Setting] を開き、左側のメニューの [SSH and GPG Key] を選択。右上の [New SSH key] をクリック
- [Title] → 後で見たときにどのパソコンで発行したキーなのかがわかる名前に。(ex. GMKのミニPC)
- [Key type] → [Authentication Key] を選択
- [Key] → コピーした公開鍵をペースト
- 入力が完了したら [Ass SSH key] をクリック

正しく設定できたのかを確認する

- `ssh -T git@github.com`
- コマンドの解説
    
    このコマンドは、設定したSSH鍵を使って**GitHubとの接続が正しく行えるかを確認する「テストコマンド」**です。
    
    新しくSSH鍵を作成してGitHubに登録したあと、実際にGit操作（pushやpull）をする前に、ちゃんと通信できるかをチェックするために使います。
    
    ---
    
    ### **コマンドの各部分の意味**
    
    - **`ssh`**
        - SSH（セキュアな通信）を使ってサーバーに接続する命令です。
    - **`T`**
        - 「対話的な端末（ターミナル）を割り当てない」という指示です。
        - GitHubのサーバーは、SSHで接続しても操作画面（シェル）を貸してくれません。このオプションを付けることで、「操作したいわけじゃなくて、**接続できるか確認したいだけです**」と伝えています。
    - **`git@github.com`**
        - 接続先のアドレスです。GitHubの場合、ユーザー名は必ず `git` になります。
- “Are you sure you want to continue connecting (yes/no/[fingerprint])?” → yes と入力
- 続いて先ほど設定したパスフレーズを入力する
- “Hi ユーザー名! You’ve successfully authenticated” と表示されたら成功
    - “Permission denied” と表示されたら失敗。手順をやり直す。

この先ターミナル操作中にパスフレーズを聞かれたら、この値を入力すればよい。
GitHubのパスワードと混乱しがちだが、ターミナルでGitHubのパスワードを入力する機会はない。

## 基本

### 場所の名前

ローカルリポジトリでは３つの場所を使い分けてファイルの状態を管理する。

- ワークツリー
    - 編集の開始地点
    - 「untracked」← 新規ファイルなどはGitの管理下に置かれていない。まずステージングする必要がある。
    - 「unmodified」← 最後のコミットから変更されていない状態。
    - 「modified」 ← なんらかの編集をすると変更済みとなる。
- ステージングエリア
- Gitディレクトリ
    - コミットした時点の変更箇所データが格納される

## 基本操作

### ディレクトリとファイルを作成

- `mkdir ディレクトリ名`
- `touch ファイル名`

### ローカルリポジトリを作る

ローカルリポジトリを作りたいディレクトに移動して作る。ls -a で.gitディレクトリがあることを確認。

- `cd ディレクトリ名`
- `git init`
- `ls -a`

### ステージングエリアに登録

逐一 git status で状態を確認。

- `git add ファイル名`
    - `git add .` ← カレントディレクトリは以下のすべてのファイルを追加
    - `git add ディレクトリ名` ← ディレクトリは以下のすべてのファイルを追加
    - `git add ディレクトリ名/ファイル名` ← 指定したファイルを追加
- `git status`

### 差分の確認

文末の改行でさえ差分となる。
ワークツリーとステージングエリアの差分

- `git diff`

ステージングエリアとGitディレクトリ（前回のコミット）の差分

- `git diff --cached`

### コミットする

コミットメッセージを複数行書く場合

- `git commit`
    - VSCodeが開く。一行目にコミットタイトルを書き、３行目以降に詳細を書く。保存してファイルを閉じればOK。

コミットメッセージが一行の時

- `git commit -m "メッセージ内容"`

ステージングエリアへの追加とコミットを一度に行う。追加対象となるのはGitの管理下に置かれているファイルの変更。-aと-mの合体。

- `git commit -am "メッセージ内容"`

### ローカルリポジトリでの操作の取り消し

ワークツリーでの変更を取り消したいとき、git restoreコマンドを使えば、直前のコミットの状態に戻すことができる。

- `git restore ファイル名`

git restoreコマンドの —stagedオプションを使うと、ステージングエリアへの間違った登録を取り消せる。

- `git restore --staged ファイル名`

### Gitの管理下にあるファイルの削除

Gitの管理下にあるファイルは、エクスプローラーなどで削除しただけでは不十分である。「削除」をステージングエリアに登録し、コミットする必要がある。
（エクスプローラーなどで）ファイル削除 → 削除したファイルをステージング → 削除のコメントつけてコミット
が流れだが、最初の「削除＋ステージング」は git rmコマンドで同時にやってくれる。

- `git rm ファイル名`
    - ディレクトリの削除の場合は-rオプションが必要。recursive(再帰的)
    - `git rm -r ディレクトリパス`
- `ls`
- `git status`
- `git commit -m "メッセージ内容"`

### Gitで管理しないファイルの設定

ログファイルやバックアップファイル、パスワードが書かれたファイルはGitで管理すべきではない。リモートリポジトリを共有しているメンバーに公開されるということを念頭に置く。

- 「.git」ディレクトリと同じディレクトリに「.gitignore」ファイルを作成する。
- .gitignoreファイルに無視したいファイル名やディレクトリ名を書く。
- `git add .gitignore`
- `git commit -m ".gitignoreファイルを追加する"`

### コミットの履歴を確認

git logコマンドを実行すると、新しい順で「コミットハッシュ」「誰がコミットをしたか」「コミット日時」「コミットメッセージ」が表示される。-pオプションを付けるとファイルの差分も確認できる。

- `git log`
- `git log -p`

## フォークとクローン

### フォーク

リモートリポジトリ間でのやり取り。他人のアカウントのリポジトリを自分の(GitHub)アカウントに複製することをフォークという。自分のアカウントに持ってこれば、オリジナルに影響を与えることなくファイルの編集ができる。一般的には共同開発をしたいリポジトリをフォークして、こちら側で変更を加えた後、最終的にフォーク元のオリジナルへその変更を反映させるという使い方をする。

- フォークしたいリポジトリを表示した状態で、右上にある [Fork] をクリック。
- [Owner] が自分のアカウント名になっているか確認
- [Repository name] と [Description] は変更したければ編集
- コピーするブランチのチェックボックスはチェックしたまま
- [Crate fork] をクリック

### クローン

リモートリポジトリをローカルリポジトリとしてコピーする操作をクローンという。最初にリモートリポジトリを特定するためのURLをGitHub上で取得する。実行したディレクトリにクローンしたディレクトリが作られる。

- クローンしたいリモートリポジトリを開き、[Code]をクリック
- [SSH]タブにあるURLをコピー
- `git clone 貼り付けたURL`
    - [shift]+[Insert] キーで貼り付けができる

※復習：ローカルリポジトリを作るには git init をしていた。git clone コマンドはもう一つの作成方法なのだ。

リモートリポジトリの設定を確認してみる。

- `git remote -v`
    - origin という文字のあとにクローン元のURLが表示されている
    - origin はリモートリポジトリの名前だが、変えることもできる

### ローカルリポジトリをGitHubにアップする

[ローカルで作業した git リポジトリを GitHub にアップする手順](https://hirofurukawa.com/push-local-repos-to-github/)

### ブランチ

ブランチはGitで記録する履歴を枝分かれさせるための機能。作業内容ごとに別々に管理できる。

### ブランチを作成して切り替える

- `git branch ブランチ名`
- `git branch` ← ブランチの確認
- `git switch ブランチ名`

作成とスイッチはgit switchの-cオプションで同時に行うことができる。

- `git switch -c ブランチ名`
    - `git checkout -b ブランチ名` でも同じことができる

### ブランチ同士の差異の確認

今使っているブランチとほかのブランチを比較できる。

- `git diff ブランチ名`
    - `git diff main`

### mainブランチにトピックブランチの変更を反映させる（ローカル内）

まずmainブランチに移動して、マージを行う。

- `git switch main`
- `git merge ブランチ名`

### プルリクエストの作成（リモートリポジトリが絡む）

作成したブランチの取り込み（マージ）を依頼するのがプルリクエスト。流れとしては、まずローカルのブランチの状態をリモートに「プッシュ」することで、リモートリポジトリにも同じブランチが作成される（コピーした）。このリモートのブランチを、リモートのmainブランチに反映させたいのだが、反映させていいのかどうか議論したい（レビューしてもらいたい）。その議論のリクエストがプルリクエストなのである。プルリクエストの作成はGitHub上で行う。リクエストするブランチを選択するわけだ。

- `git push origin ブランチ名`
- GitHubを開き、[Pull request] から[New pull request] を選択
- 左のベースブランチに自分のアカウントのリモートリポジトリを選択
- 右ではレビュー対象のブランチを選択する
- タイトルとコメントを書く。変更内容や背景をマークダウンで書き、わかりやすくする。
- レビューをお願いしたい場合は右の[Reviewers]でユーザー名を選択する
- [Create pull request] で作成完了

### ブランチの削除

ローカルリポジトリのブランチだけでなく、リモートリポジトリのブランチも削除できる。

ローカルリポジトリのマージ済みのブランチの削除

- `git branch --delete ブランチ名`
    - `git branch -d ブランチ名` でもOK

ローカルリポジトリのマージ状況に関わらずブランチを削除

- `git branch -D ブランチ名`

リモートリポジトリのブランチの削除

- `git push --delete origin ブランチ名`
    - `git push origin :ブランチ名` でもOK

### ローカルリポジトリのmainブランチを最新化する

リモートリポジトリの最新状態をローカルリポジトリに反映させる際は、まずローカルリポジトリをmainブランチに切り替えてからpullする。

- `git switch main`
- `git pull origin main`

### トピックブランチに最新のmainブランチを反映させる（ローカル）

まず反映させたいトピックブランチに移動してからマージする

- `git switch ブランチ名`
- `git merge main`

### コンフリクトの発生

まずgit statusで確認

- `git status`
- 「both modified」→ マージ先とマージ元の両ブランチで変更を加えた
- 「you」→ マージ元である現在使用中のブランチのこと
- 「them」→ マージ先のブランチのこと
- 「deleted by them」→ 使用中のブランチで編集したファイルがマージ先のブランチで削除されていた
- プルリクエスト中だった場合、githubにもコンフリクトメッセージがでてマージできなくなっているが、ローカルで解消してプッシュすれば解消される。

VSCodeでは以下のようになっている。

<<<から>>>までの行がコンフリクトしている箇所、===の行がマージ先とマージ元の変更箇所の境目。２つを融合したものに変更する。>>>とかも消す。

- `ファイルの編集`

編集が完了したらステージングエリアに追加する。git statusでファイルの編集を確認。

- `git add -A`
- `git status`

All conflicts fixed とある。コンフリクトにより失敗したマージは、解消してコミットを行うまでマージしている最中（マージが未完）であるとみなされる。パラメーターなしでコミットすれば完了。コミットメッセージが入力された状態でエディターが立ち上がるが、何も編集せずに閉じればよい。

- `git commit`

解消したらプッシュする

- `git push origin ブランチ名`

プルリクエストでコンフリクトメッセージが出ていた場合は、これにてマージができるようになっている。
